java基础八股速记.md

Created2025-12-20|Updated2025-12-24|学习codingjava语言基础

Java程序的执行过程

.java文件 -> 编译器(javac) -> .class字节码 -> JVM(解释执行/JIT编译执行) -> 机器码 -> 操作系统执行

这个过程涉及的核心组件：编译器、JVM、JIT (Just-In-Time) 和 AOT (Ahead-Of-Time)。

字节码

定义：字节码是由.java或者其他支持字节码编译器的语言编译后形成的代码 (.class 文件)。

存在的原因/解决的问题：字节码不是针对特定 OS 和 ISA (指令集架构) 的，它可以在任意平台上的 JVM 中运行，实现了 “Write Once, Run Anywhere” 的跨平台特性。

特性：

  • 混合执行模式：
    • 解释执行：JVM 解释器逐行将字节码翻译成机器码执行，启动快但运行慢。
    • JIT 编译：JVM 将热点代码（HotSpot）编译成本地机器码并缓存（Code Cache），后续直接执行机器码，效率接近 C++。但是热点代码是如何检测的？TODO
  • JVM 本质：JVM 本质上是运行在 OS 上的软件，它负责解析字节码并调用底层的 OS 指令。

问题：

  • 既然编译型更快，为什么还要采用解释型？
    • 启动响应：解释器立即执行，无需等待编译时间。
    • 内存占用：JIT 编译后的机器码需要占用内存（Code Cache）。
    • 优化：混合模式权衡了启动速度和峰值性能。
  • **AOT (Ahead-Of-Time)**：在程序运行前直接将 Java 代码编译成机器码（如 GraalVM），解决启动慢和预热问题，但丧失了部分动态特性（如动态加载类）。

类与接口

**类 (Class)**：创建对象的模板，定义了属性（状态）和行为（方法）。支持继承（Is-A）。
**接口 (Interface)**：行为的规范/契约（Can-Do）。支持多实现。

类和接口的实质：

  • 在内存中，类和接口本质上都对应一个 java.lang.Class 对象。
  • Class.isInterface() 可用于区分。

构成对比：

  • 类：元数据、字段、方法、构造器、静态代码块。
  • 接口：
    • JDK 7及以前：只能有 public abstract 方法和 public static final 常量。
    • JDK 8：增加 default 方法（默认实现，便于接口升级兼容）和 static 方法。
    • JDK 9：增加 private 方法（供接口内部默认方法复用）。

问题：

  • 抽象类和接口的区别？
    • 设计层面：抽象类是 “Is-A” (它是什么)，用于代码复用（模板方法模式）；接口是 “Has-A/Can-Do” (它能做什么)，用于解耦和规范行为。
    • 语法层面：类单继承，接口多实现；抽象类可以有构造器和状态（成员变量），接口一般只有常量。
    • 场景：设计支付渠道，AbstractPaymentChannel 可以处理通用的日志、校验逻辑；PaymentInterface 定义 pay(), refund() 方法规范。

对象 (Object)

定义：Java 中所有类的父类（根类）。任何类如果未显式继承其他类，默认继承 Object。

构成：从JVM角度来说对象的构成

机制：对象初始化的机制（JVM）

核心方法：

  • equals(Object obj): 判断对象是否相等（默认比较地址，通常需要重写比较内容）。
  • hashCode(): 返回对象的哈希码。约定：equals 相等，hashCode 必须相等；hashCode 相等，equals 不一定相等（哈希冲突）。
  • toString(): 返回对象的字符串表示。
  • clone(): 创建并返回对象的副本（浅拷贝，需实现 Cloneable 接口）。
  • getClass(): 返回对象的运行时类。
  • wait(), notify(), notifyAll(): 线程通信/同步相关。
  • finalize(): 已废弃（JDK 9），垃圾回收前被调用，不可靠。

问题：

  • == 和 equals 的区别？
    • ==：比较基本类型是比较值；比较引用类型是比较内存地址。
    • equals：默认同 ==，但 String、Integer 等重写了该方法，比较的是值/内容。
  • 为什么重写 equals 必须重写 hashCode？
    • 如果只重写 equals，可能导致两个“逻辑相等”的对象有不同的 hashCode。在 HashMap/HashSet 中，这将导致无法正确通过 key 找到 value（因为先算 hash 确定桶下标）。

包装类

定义：8种基本数据类型对应的引用类型（如 int -> Integer, char -> Character）。为了让基本类型也能参与面向对象的操作（如泛型、集合中只能存对象）。

特性：

  • 自动装箱/拆箱：编译器自动调用 Integer.valueOf(int) 和 Integer.intValue()。
  • **缓存机制 (享元模式)**：
    • Byte, Short, Integer, Long 默认缓存 [-128, 127] 的对象。
    • Character 缓存 [0, 127]。
    • 坑：Integer a = 100; Integer b = 100; (a==b 为 true)；Integer c = 200; Integer d = 200; (c==d 为 false，超出缓存范围创建新对象)。

问题：

  • 所有 POJO 属性为什么推荐用包装类？
    • 因为数据库的查询结果可能是 null。基本类型（如 int）默认值是 0，无法区分“数据不存在”和“数据值为0”。

String (字符串)

定义：Java 中最常用的类，代表字符串。

特性：

  • **不可变性 (Immutable)**：
    • 底层是 private final char[] (JDK 8) 或 byte[] (JDK 9+，节省内存)。
    • 一旦创建，内容无法修改。修改 String 其实是创建了新对象。
    • 优点：线程安全；支持字符串常量池（节省内存）；Hash 值可缓存（适合做 Map 的 key）。
  • 字符串常量池：
    • String s = "abc"：若池中有则直接返回引用，无则创建并放入池。
    • new String("abc")：在堆中创建对象，且确保池中也有 “abc”。

辅助类：

  • StringBuilder：可变，线程不安全，效率高。
  • StringBuffer：可变，线程安全（方法加了 synchronized），效率低。

问题：

  • String s = new String(“a”) + new String(“b”) 创建了几个对象？
    • 涉及 StringBuilder、常量池中的 “a”, “b” 以及堆中的实例，情况较复杂，重点在于理解拼接底层是 StringBuilder。

集合 (Collections)

体系结构：

  • Collection (接口):
    • List (有序，可重复): ArrayList, LinkedList, Vector.
    • Set (无序，唯一): HashSet, TreeSet, LinkedHashSet.
    • Queue: PriorityQueue, ArrayDeque.
  • Map (键值对，Key 唯一): HashMap, TreeMap, Hashtable, ConcurrentHashMap.

核心实现：

  • ArrayList:
    • 底层：动态数组 Object[]。
    • 特点：随机访问快 (O(1))，插入删除慢 (涉及扩容和拷贝)。
    • 扩容：默认初始容量 10，扩容为原来的 1.5 倍。
  • LinkedList:
    • 底层：双向链表。
    • 特点：插入删除快，随机访问慢。
  • HashMap:
    • 底层：数组 + 链表 + 红黑树 (JDK 8)。
    • put 流程：计算 Key 的 hash -> 确定数组下标 -> 碰撞处理（链表长度 > 8 且数组长度 > 64 转红黑树）。
    • 扩容：负载因子 0.75，每次扩容 2 倍。需要重新哈希 (rehash)。
    • 线程安全：不安全。多线程扩容可能死循环 (JDK 7) 或数据丢失 (JDK 8)。
  • ConcurrentHashMap:
    • JDK 7: 分段锁 (Segment)，锁粒度较大。
    • JDK 8: Node数组 + CAS + synchronized (锁桶头结点)，锁粒度更细，并发度高。

泛型 (Generics)

定义：参数化类型。在编译时检查类型安全，避免强制类型转换。

特性：

  • **类型擦除 (Type Erasure)**：Java 泛型是伪泛型。在编译后的字节码中，泛型信息会被擦除（通常变为 Object 或上限类型），运行时 JVM 不知道泛型具体类型。
    • 这也是为什么 List<String> 不能赋值给 List<Object> (不协变)，也不能 new T()。
  • 通配符：
    • ? extends T (上界): 只能读，不能写 (除了 null)。适用于生产者 (Producer)。
    • ? super T (下界): 只能写，读取出来是 Object。适用于消费者 (Consumer)。
    • PECS 原则: Producer Extends, Consumer Super.

异常 (Exception)

体系结构：

  • Throwable (根)
    • Error: 严重错误，JVM 无法恢复 (如 OutOfMemoryError, StackOverflowError)。
    • Exception:
      • Checked Exception (受检异常): 编译期强制处理 (try-catch 或 throws)。如 IOException, SQLException。业务逻辑中可预期的异常。
      • Unchecked Exception (运行时异常/RuntimeException): 编译期不检查。如 NullPointerException, IndexOutOfBoundsException。通常是代码逻辑错误。

关键语法：

  • try-catch-finally: finally 块通常一定会执行（除非 System.exit）。
  • try-with-resources (JDK 7): 自动关闭实现 AutoCloseable 接口的资源 (如 IO 流)，避免内存泄漏。

反射 (Reflection)

定义：在运行状态中，动态获取类的信息（属性、方法）并操作对象的能力。

机制：通过 Class 对象（字节码文件）反向获取类的元数据，反射的过程是线程安全的（loadClass使用synchronized锁，设置缓存使用了CAS）。反射机制的流程

  • 构造器获取对象：获取构造器的过程主要是先从缓存**(软引用)**中查，查不到就去JVM中加载，再调用构造器的newInstance方法
  • 方法调用过程：无论是实例方法还是构造方法，最终都是调用JVM中的C/C++方法

核心的对象和方法：Class对象及其获取方法，Field对象，Method对象，Constructor对象。

用途：

  • 框架核心（Spring IOC/AOP）：通过配置动态加载类。
  • 动态代理。
  • JDBC 加载驱动。

优缺点：

  • 优点：极高的灵活性，动态性。
  • 缺点：性能开销大（解释操作，JVM 无法优化）；破坏封装性（可强制访问 private 成员）。

注解 (Annotation)

定义：本质是个接口，代码里的特殊标记，为代码提供元数据。本身不产生逻辑，需要结合反射或编译器使用。

机制：由于是个接口，所以注解中的属性实质上是public abstarct修饰的抽象方法，打上注解的类会根据传的值默认实现这个方法。

分类：

  • 标准注解：@Override, @Deprecated, @SuppressWarnings.
  • 元注解 (修饰注解的注解):
    • @Target: 作用范围 (类, 方法, 字段等)。
    • @Retention: 生命周期 (SOURCE-编译丢弃, CLASS-字节码中保留, RUNTIME-运行时保留/反射可读)。
    • @Documented, @Inherited.

场景：

  • Spring: @Controller, @Service, @Autowired.
  • Lombok: @Data (编译期通过注解处理器修改 AST 生成代码).

详见：https://www.cnblogs.com/ziph/p/13056092.html

静态方法 (static)

定义：由 static 关键字修饰的方法，属于类而不属于对象。

特性：

  • 随着类的加载而加载，内存中只有一份。
  • 可以直接通过 类名.方法名() 调用。
  • 限制：静态方法中不能使用 this 或 super，不能直接访问非静态成员（因为非静态成员依赖对象实例，而静态方法执行时对象可能未创建）。

场景：

  • 工具类：Math.max(), Collections.sort()。状态无关，不需要实例化。
  • 单例模式：getInstance()。

JVM 简单速记 (补充)

**内存区域 (Runtime Data Area)**：

  • 线程私有：
    • 程序计数器：当前线程执行的字节码行号。
    • 虚拟机栈：Java 方法执行的栈帧（局部变量表、操作数栈等）。异常：StackOverflowError。
    • 本地方法栈：Native 方法。
  • 线程共享：
    • **堆 (Heap)**：存放对象实例。GC 的主要区域。异常：OOM。
    • **方法区 (Method Area)**：存放类信息、常量、静态变量。实现：永久代(JDK 7) -> 元空间(JDK 8, 使用本地内存)。

类加载机制：

  • 过程：加载 -> 验证 -> 准备 -> 解析 -> 初始化。
  • 双亲委派模型：
    • Bootstrap (启动类加载器) <- Extension (扩展) <- Application (应用)。
    • 机制：收到请求先委派给父类加载，父类找不到自己才加载。
    • 目的：安全（防止篡改核心库），避免重复加载。
    • 打破：Tomcat, SPI 机制。

代理 (Proxy)

定义：为其他对象提供一种代理以控制对这个对象的访问。

分类：

  • 静态代理：
    • 编译期确定代理类。一个代理类对应一个目标类，代码冗余。
  • 动态代理 (运行时生成)：
    • JDK 动态代理：基于接口。利用反射机制生成一个实现代理接口的匿名类，在调用具体方法前调用 InvokeHandler 来处理。生成proxy的三个参数：ClassLoader，要代理类的实现的接口数组，以及InvocationHandler
      • 流程是一个类实现若干个接口，然后自己实现InvocationHandler接口或者直接传InvocationHandler,该类的所有实现接口以及该类的类加载器。
      • 核心对象是修改JVM加载进内存的字节码，生成$proxy对象，由$proxy实现。源码中$proxy继承了proxy并实现传入的所有接口，调用接口的方法实质上都是调用proxy的invoke方法。
      • 详见：https://juejin.cn/post/6974018412158664734
    • CGLIB 动态代理：基于继承。生成的proxy是继承被代理类。利用 ASM 开源包，对代理对象类的 class 文件加载进来，通过修改其字节码生成子类来处理。在此之前也要实现MethodInterceptor接口。(待填坑)
      • 特性是继承+FastClass机制

区别/选择：

  • JDK 动态代理只能代理实现了接口的类。
  • CGLIB 可以代理未实现接口的类（但不能代理 final 类）。

场景：

  • Spring AOP (面向切面编程) 的核心。
  • RPC 框架 (远程过程调用)。
  • 事务控制、日志记录、权限校验。

IO (Input/Output)

体系结构：

  • Unix IO

    • 分类：阻塞式，非阻塞式，IO复用（select、poll或者epoll），信号驱动式IO，异步IO（AIO）。
    • 涉及的系统调用：recvfrom

  • **BIO (Blocking IO)**：同步阻塞。传统的 java.io 包。一个连接一个线程。

  • **NIO (Non-blocking IO)**：同步非阻塞。java.nio 包。基于 Channel (通道), Buffer (缓冲区), **Selector (选择器)**。

    • Selector：多路复用器，一个线程可以管理多个 Channel。

  • **AIO (Asynchronous IO)**：异步非阻塞。java.nio.channels.AsynchronousSocketChannel。基于回调机制。

场景：

  • BIO：连接数目小且固定的架构。
  • NIO：连接数目多且连接比较短（轻操作）的架构。如：聊天服务器，Tomcat (默认)。
  • AIO：连接数目多且连接比较长（重操作）的架构。

问题：

  • select, poll, epoll 的区别？ (底层 OS 机制)
    • select/poll：轮询所有连接，效率随连接数增加而下降。
    • epoll：基于事件驱动 (callback)，只处理活跃连接，效率高。NIO 在 Linux 上底层使用 epoll。

序列化 (Serialization)

定义：

  • 序列化：将对象转换为字节序列的过程。
  • 反序列化：将字节序列恢复为对象的过程。

机制：

  • 实现 java.io.Serializable 接口 (标记接口，无方法)。
  • serialVersionUID：版本号。若反序列化时 ID 不一致，抛出 InvalidClassException。
  • transient 关键字：修饰的变量不会被序列化 (如密码等敏感字段)。

**替代方案 (序列化协议)**：

  • JDK 原生序列化：性能差，体积大，不仅跨语言。
  • JSON (Jackson, Gson): 可读性好，跨语言，但体积较大。
  • Protobuf (Google): 二进制，性能高，体积小，跨语言，需定义 .proto 文件。
  • Hessian: 二进制，支持跨语言，RPC 框架常用。

坑：

  • 反序列化时不会调用构造方法。
  • 单例模式若被序列化/反序列化，会破坏单例 (需实现 readResolve 方法)。
